SimGrid 是一个分布式应用模拟器开发框架。本手册对 SimGrid 4.1 官方文档进行完整中文本地化。

教程 1：模拟分布式算法

SimGrid 最初就是为了研究分布式算法而设计的。它的 S4U 接口可以用来评估云计算、P2P、HPC、IoT 等场景中的算法行为。一个典型的 SimGrid 模拟由多个 Actor 组成：每个 Actor 在某个模拟 Host 上运行用户函数，并通过 S4U 明确表达计算、通信、磁盘访问等 Activity。这些活动发生在 Host、Link、Disk 等资源上，SimGrid 负责预测活动耗时，并据此推进等待中的 Actor。

Actor 之间不会直接互相发送消息，而是把消息投递到 Mailbox。Mailbox 是通信 Actor 的会合点：发送方把数据放入邮箱，接收方从同名邮箱取出数据。本教程以一个简单但完整的 Master/Workers 应用为主线：master 分发任务，workers 执行任务。示例初始版本使用轮询调度，后续实验会逐步扩展它。

认识 Master/Workers

这个示例应用包含两类 Actor：master 负责向一组 workers 分发计算任务；workers 收到任务后在自己所在的模拟主机上执行。初始代码按 round-robin 顺序分发任务，也就是依次把任务发给每个 worker，直到所有任务发完。

Actor 代码

master 函数至少需要 3 个参数：任务数量、每个任务的计算量（flop 数）、每个任务的通信量（字节数）。第三个参数之后的所有参数都被解释为 worker 所在主机的名字。master 会根据这些主机名取得同名 Mailbox，然后逐个发送任务。任务发完后，master 再给每个 worker 发送一个计算量为负数的任务，作为“停止工作”的约定信号。

这个例子里最核心的 SimGrid API 是 simgrid::s4u::Mailbox::by_name() 和 simgrid::s4u::Mailbox::put()。日志建议使用 XBT_INFO，它会把模拟时间和 Actor 名称一起打印出来，比直接用 printf() 或 std::cout 更适合分析模拟输出。

Master（C++）

static void master(std::vector<std::string> args) {
  long tasks_count = std::stol(args[1]);
  double compute_cost = std::stod(args[2]);
  long communication_cost = std::stol(args[3]);
  std::vector<sg4::Mailbox*> workers;
  for (unsigned int i = 4; i < args.size(); i++)
    workers.push_back(sg4::Mailbox::by_name(args[i]));
  for (int i = 0; i < tasks_count; i++)
    workers[i % workers.size()]->put(new double(compute_cost), communication_cost);
  for (auto* m : workers) m->put(new double(-1.0), 0);  // 终止信号
}

Worker（C++）

static void worker(std::vector<std::string> args) {
  sg4::Mailbox* mailbox = sg4::Mailbox::by_name(sg4::this_actor::get_host()->get_name());
  double cost;
  do { auto msg = mailbox->get_unique<double>(); cost = *msg;
       if (cost > 0) sg4::this_actor::execute(cost);
  } while (cost > 0);
}

worker 以自己所在 Host 的名字作为邮箱名，循环等待任务。收到正数计算量时，它调用 simgrid::s4u::this_actor::execute() 消耗对应的模拟计算时间；收到负数计算量时，循环结束，Actor 退出。Python 版本的结构相同：读取命令行参数、通过 Mailbox.by_name() 找邮箱、用 put()/get() 传任务、用 this_actor.execute() 表达计算。

启动模拟

main() 中创建 Engine，把可被部署文件引用的 Actor 函数注册进去，再加载平台文件和部署文件，最后调用 run()。部署文件中的 function="master"、function="worker" 会和这里注册的函数名对应起来。

int main(int argc, char* argv[]) {
  sg4::Engine e(&argc, argv);
  e.register_function("master", &master);
  e.register_function("worker", &worker);
  e.load_platform(argv[1]); e.load_deployment(argv[2]); e.run();
}

运行时通常把平台文件作为第一个参数、部署文件作为第二个参数：

./master-workers platform.xml deployment.xml

平台文件

平台文件描述要模拟的硬件环境：主机、链路和路由。下面的片段定义了两个 Host、一条 Link，以及两台主机之间经过该链路的路由。Host 的 speed 表示计算速度，Link 的 bandwidth 和 latency 表示通信能力。

<zone id="zone0" routing="Full">
  <host id="Tremblay" speed="98.095Mf"/>
  <host id="Fafard" speed="76.296Mf"/>
  <link id="6" bandwidth="41.28MBps" latency="59.904us"/>
  <route src="Tremblay" dst="Fafard"><link_ctn id="6"/></route>
</zone>

真实教程中的平台文件还会包含更多主机、链路和路由。重点是：算法代码不需要硬编码机器数量和网络拓扑，模拟规模和结构由平台文件控制。

部署文件

部署文件说明哪些 Actor 在哪些 Host 上启动，以及传给 Actor 函数的参数。下面的 master Actor 收到 20 个任务、每个任务 50,000,000 flop、每个任务传输 1,000,000 字节，并把 worker 主机名作为后续参数。

<actor host="Tremblay" function="master">
  <argument value="20"/><argument value="50000000"/><argument value="1000000"/>
  <argument value="Tremblay"/><argument value="Fafard"/>
</actor>
<actor host="Tremblay" function="worker"/>

完整部署中，每个 worker 也会作为独立 Actor 启动。最初的做法要求 master 参数中列出所有 worker 主机名，这很直观，但扩展到大平台时会显得笨重，后续实验会逐步消除这种手工配置。

执行示例

运行后，日志会显示 master 按顺序发送任务、worker 收到并执行任务、最后收到终止消息。日志前缀里的数字是模拟时间，不是墙钟时间。也就是说，即使真实运行只花几毫秒，输出里的时间仍代表算法在模拟平台上的预测执行时间。

[Tremblay:master:(1) 0.000000] [xbt/INFO] Got 2 workers and 20 tasks to process
[Tremblay:worker:(2) 0.024348] [xbt/INFO] Processing task
[Fafard:worker:(3) 0.048696] [xbt/INFO] Processing task
[Tremblay:master:(1) 8.934521] [xbt/INFO] All tasks have been dispatched. Request workers to stop.

自己动手改进

官方教程接下来要求你在已有 Master/Workers 示例上动手改。推荐先确认本地环境能编译和运行提供的代码，再逐个实验修改。教程同时支持 C++ 和 Python，下面的说明按通用流程组织。

使用 Docker

如果不想在本机安装 SimGrid，可以使用官方提供的 Docker 镜像。Docker 方式的优点是环境一致、依赖完整；缺点是编辑器、文件映射和调试体验可能不如本机安装顺手。典型流程是在容器中进入示例目录，编译并运行 Master/Workers。

docker run --rm -it -v "$PWD:/source" simgrid/stable bash
cd /source
cmake -S . -B build
cmake --build build
./build/master-workers platform.xml deployment.xml

本机原生环境

如果本机已经安装 SimGrid，则可以直接使用系统编译器和 SimGrid 库。C++ 项目通常通过 CMake 查找 SimGrid；Python 示例则需要可导入 simgrid 包。原生方式更适合反复改代码、看日志和调试。

cmake -S . -B build
cmake --build build
./build/master-workers platform.xml deployment.xml

python3 master-workers.py platform.xml deployment.xml

熟悉提供的代码

官方示例通常按实验阶段提供多个版本。你应该先定位初始 Master/Workers 文件、平台文件、部署文件和构建脚本，确认它们能跑通。之后每个 Lab 都在前一个版本基础上改进：先改部署方式，再改 worker 创建方式，再控制实验时长，最后做平台变化和竞争应用分析。

实验 1：更简单的部署

初始版本要求 master 在部署文件里显式列出每个 worker 所在 Host 的名字。这种方式可读，但一旦主机很多就很难维护。实验 1 的目标是简化部署：worker 启动后向一个公共服务登记自己，master 不再通过部署参数拿 worker 列表，而是通过这个登记信息找到可用 worker。

一种常见写法是引入一个“黄页”式的注册表：worker 用自己的 ID 或邮箱名注册，master 启动后读取注册表并分发任务。这样部署文件只需要声明 master 和 workers，不需要把所有 worker 名字重复塞进 master 参数。

// worker 启动时登记自己的邮箱
auto* mailbox = sg4::Mailbox::by_name(sg4::this_actor::get_host()->get_name());
// master 之后从登记表取得所有 worker mailbox，而不是解析一长串部署参数

小结

完成实验 1 后，部署文件更短，master 和 worker 的耦合更低。你也会更清楚地区分两件事：部署文件负责“在哪里启动哪些 Actor”，应用内部逻辑负责“这些 Actor 如何发现彼此”。

实验 2：使用整个平台

实验 2 进一步减少手工配置：不再在部署文件里逐个声明所有 worker，而是在 master 内部遍历平台上的所有 Host，并在每台合适的 Host 上动态创建 worker。这样同一份应用代码可以直接套用到不同规模的平台文件上。

由 master 创建 workers

master 可以通过 Engine::get_all_hosts() 获取平台中的所有 Host，再通过 Actor::create() 在这些 Host 上启动 worker。通常要避免把 master 自己所在的 Host 重复当作普通 worker，或者根据实验需求选择是否允许 master 所在 Host 也承担 worker 任务。

auto hosts = sg4::Engine::get_instance()->get_all_hosts();
for (auto* host : hosts) {
  if (host != sg4::this_actor::get_host())
    sg4::Actor::create("worker", host, worker);
}

Master-Workers 通信

动态创建 worker 后，通信命名也要跟着调整。最简单的策略仍然是“每个 worker 使用 Host 名作为 Mailbox 名”，master 根据 Host 列表生成同名 Mailbox。更灵活的策略是创建 worker 时传入 worker ID，让 worker 和 master 都使用这个 ID 构造邮箱名。

std::string mailbox_name = "worker-" + std::to_string(i);
sg4::Actor::create("worker-" + std::to_string(i), host, worker, mailbox_name);
workers.push_back(sg4::Mailbox::by_name(mailbox_name));

小结

实验 2 之后，平台文件真正变成了控制实验规模的入口。换一个更大的 XML 平台，程序就能自动创建更多 workers，而不需要同步修改部署文件。

实验 3：固定实验时长

前两个版本通常以“任务全部完成”为结束条件。实验 3 改成固定模拟时长：master 不断产生任务，直到模拟时钟达到指定时间，然后停止分发并关闭 workers。这类设置更适合评估吞吐量、资源利用率或长期运行服务。

关键 API 是 Engine::get_clock()。master 可以在循环中检查当前模拟时间，超过截止时间后发送终止消息。如果某些 Actor 仍在等待或执行，也可以根据实验语义使用 Actor::kill() 主动结束。

double deadline = 1000.0;
while (sg4::Engine::get_clock() < deadline) {
  workers[next_worker]->put(new double(compute_cost), communication_cost);
  next_worker = (next_worker + 1) % workers.size();
}
for (auto* worker : workers)
  worker->put(new double(-1.0), 0);

控制日志详细程度

实验时间变长后，日志很容易变得巨大。SimGrid 使用 XBT 日志系统，可以通过命令行控制不同组件的日志级别。调试时打开详细日志，批量实验时降低到 warning 或 critical，能让输出更稳定、更易分析。

./master-workers platform.xml deployment.xml --log=root.thres:info
./master-workers platform.xml deployment.xml --log=master.thres:debug
./master-workers platform.xml deployment.xml --log=root.thres:warning

实验 4：What-if 分析

SimGrid 的一个核心价值是做 what-if 分析：不改算法代码，只改变平台能力、链路状态或故障模式，观察算法表现如何变化。实验 4 分别从计算速度、通信速度和故障处理三个角度扩展示例。

计算速度

Host 的计算速度可以在平台文件中直接设置，也可以通过 profile 描述随时间变化的速度。例如某台机器在一段时间内降频，就会影响其上 worker 的任务完成速度。应用侧仍然只表达“我要执行多少 flop”，模拟器会根据当前 Host speed 推导耗时。

<host id="worker-1" speed="100Mf">
  <prop id="wattage_per_state" value="100:200"/>
</host>

// C++ 中也可为 Host 设置速度变化 profile
host->set_speed_profile(profile);

通信速度

同理，Link 的带宽和时延会直接影响 Mailbox::put() 对应通信活动的完成时间。你可以通过平台文件改变链路带宽，或使用 bandwidth profile 模拟网络拥塞、限速、维护窗口等情况。

<link id="backbone" bandwidth="125MBps" latency="100us"/>

// 让链路带宽随时间变化
link->set_bandwidth_profile(profile);

处理故障

平台资源也可以失效。Host 或 Link 故障会让正在进行的活动失败，应用代码需要捕获相关异常，并决定重试、跳过任务、重新分配任务或终止实验。这个实验的重点不是写出最完美的容错算法，而是观察“同一个算法在故障平台上的行为”。

try {
  mailbox->put(new double(compute_cost), communication_cost);
} catch (const simgrid::NetworkFailureException& e) {
  XBT_WARN("Communication failed; rescheduling the task");
} catch (const simgrid::HostFailureException& e) {
  XBT_WARN("Worker host failed; selecting another worker");
}

实验 5：竞争应用

前面的实验都只有一个 master-worker 应用。实验 5 引入多个同时运行的应用，让它们共享同一平台资源。这样可以观察竞争：两个 master 同时向 workers 分发任务时，链路、CPU 和路由上的资源争用会改变每个应用的完成时间。

实践上，可以在部署文件中启动多个 master，也可以让一个上层 Actor 动态创建多个 master。为了区分输出，建议给不同应用设置不同名称或 tracing category。

sg4::this_actor::get_host()->set_property("application", "app-A");
activity->set_tracing_category("app-A");

// 或在部署文件中启动多个 master，传入不同参数
<actor host="Tremblay" function="master">...</actor>
<actor host="Fafard" function="master">...</actor>

可视化结果

SimGrid 可以导出 tracing 信息，用 Viva、R、Python 或其他工具可视化。竞争应用场景中，建议至少画出每个应用的任务完成时间、Host 利用率和关键 Link 负载。这样能区分“算法本身慢”和“被其他应用挤占资源”。

./master-workers platform.xml deployment.xml \
  --cfg=tracing:yes \
  --cfg=tracing/filename:trace.trf

# 后续可用 R / Python 脚本读取 trace 或日志，绘制时间线和吞吐量曲线

进一步改进

官方教程最后鼓励继续扩展这个例子。可以尝试把 round-robin 改成基于队列长度、Host 速度或历史完成时间的调度策略；可以让 worker 主动拉取任务而不是 master 主动推送；也可以加入任务优先级、失败重试、异构任务大小、批处理和背压控制。

更贴近研究场景的改进包括：把平台文件换成真实集群拓扑；把任务流替换成真实 trace；对比多种调度策略的平均完成时间、尾延迟和资源利用率；在不同网络拥塞或故障 profile 下重复实验，评估算法鲁棒性。

学完本教程之后

完成本教程后，你已经掌握了 SimGrid 分布式算法模拟的基本闭环：写 Actor、描述平台、描述部署、运行实验、解释模拟时间、改变平台条件并分析结果。下一步可以继续学习 SMPI 教程来模拟 MPI 应用，或阅读 S4U 手册深入掌握 Actor、Activity、Mailbox、Host、Link、Disk 等接口。

教程 2：形式化验证与模型检验

Mc SimGrid 是嵌入 SimGrid 的全功能模型检验器（model checker），可穷举探索分布式程序的所有可能执行路径，发现死锁、断言失败、竞态条件等 bug。

核心概念

  • 软件模型检验：不依赖数学建模，直接把程序本身当作模型来验证
  • 无状态模型检验：通过 UNIX fork 穷举所有执行分支
  • DPOR（动态偏序约简）：减少需探索的冗余路径
  • 关键转换（Critical Transition）：从此步之后所有执行都走向 bug
  • 时间在模型检验模式下变为离散的步骤（无性能时间）

两种观测方案

方案可观测事件适用
sthread（LD_PRELOAD 注入）pthread 操作、互斥锁、信号量等多线程 C/C++ 程序
SMPIMPI 通信操作MPI 程序
mclang（v4.1+，编译器插桩）内存读写事件，检测数据竞态多线程程序

运行方式

# 多线程程序（sthread 注入）
simgrid-mc --sthread ./my_program

# MPI 程序
smpirun -wrapper simgrid-mc -np 4 -platform p.xml ./my_mpi_program

# 重放出错的执行路径（获取详细回溯）
./my_program --cfg=model-check/replay:'1;2;1;1;2;4;1;1;3;1'

示例 1：哲学家就餐死锁

5 个哲学家、5 把叉子——如果所有哲学家同时拿起左手叉子，死锁。常规运行 10000 次也很难触发。用 simgrid-mc --sthread 运行，0.1 秒内就检测到死锁，并输出完整执行路径和关键转换点。

示例 2：数据竞态检测

# 用 mclang 编译（自动插桩内存访问）
mclang plusplus.c -o plusplus -g
# 输出: Successfully instrumented 4 read operations and 3 writes operations

# 模型检验
simgrid-mc --sthread ./plusplus
# 检测到: datarace at location 0x... between actor 2 and 3
# First operation was a WRITE, Second was a READ — both at plusplus.c:8

示例 3：非确定性消息接收（S4U/MPI）

Server 假设收到的最后一条消息来自 client 3。模型检验器发现消息顺序是不确定的——取决于链路速度，任何顺序都可能。通过穷举探索找出导致断言失败的执行路径。

关键配置

配置项说明
model-check/reductiondpor（默认）/ sdpor / odpor / none
model-check/max-depth最大探索深度（默认 1000）
model-check/timeout软超时（秒）
model-check/replay重放指定执行路径
model-check/dot-output输出 Graphviz dot 文件

局限性

  • 只观测已插桩的事件（MPI/线程/内存），不观测磁盘 I/O 和普通通信
  • 状态空间随 Actor 数和事件数指数增长——需精简验证程序
  • 时间为离散步骤，无法评估性能

SMPI：模拟 MPI 应用

运行未修改的 MPI 程序（C/C++/Fortran）。用 smpicc 替代 mpicc，smpirun 替代 mpirun——主要区别是多一个平台文件。

编译运行

smpicc prog.c -o prog       # C
smpicxx prog.cpp -o prog     # C++
smpirun -hostfile h.txt -platform p.xml ./prog
cmake -DMPI_C_COMPILER=/opt/simgrid/bin/smpicc .

默认 dlopen 方式（多次加载可执行文件，快）。可选 mmap（慢但更通用）。用 --cfg=smpi/privatization:no|dlopen|mmap 控制。

简介

SimGrid 是一个分布式应用在分布式平台上执行的模拟器开发框架。

四大用途

  1. 方案对比：不需真实硬件
  2. 平台协同设计：修改平台文件远比搭真实平台容易
  3. 调试真实应用：可复现的"全知视角"
  4. 形式化验证：穷举执行路径找 bug

项目组成

组件说明
应用C++/Python/C API 或 MPI 程序
模拟平台XML/C++ 硬件描述，支持动态行为
部署Actor → Host 映射
平台模型核心：精确建模并发通信的网络竞争

局限性

  • 专注分布式系统，不适用实时多线程
  • 不支持 5G/LoRa
  • 抽象掉 L2/L3 缓存、内核调度器

成功案例

  • 3,000+ 论文引用
  • Tibidabo ARM 集群建前就用 SimGrid 预测加速比
  • 调试 BigDFT、StarPU、TomP2P

C++ 平台定义

当 XML 不够灵活时（如 Fat-Tree 叶子需嵌入 CPU+GPU+内部总线），用 C++ 直接构建。

NetZone* zone = Engine::get_instance()->get_netzone_root()->add_netzone_star("z");
Link* l = zone->add_link("l","125MBps")->set_latency("24us")->seal();
Host* h = zone->add_host("h","1Gf")->seal();
zone->add_route(h, nullptr, {{l, LinkInRoute::Direction::UP}}); zone->seal();

Split-Duplex 链路

auto* link = zone->add_split_duplex_link("1","125MBps")->set_latency("24us")->seal();
zone->add_route(S1, C1, nullptr, nullptr, {{link, LinkInRoute::Direction::UP}});

Fat-Tree + 复杂叶子节点

root->add_netzone_fatTree("bob", root, 2, {4,4}, {1,2}, {1,2},
    "100Mbps","50us", Link::SharingPolicy::SPLITDUPLEX)
  ->set_netzone_cb(create_hostzone)   // 每个叶子 = CPU + GPU + 内部总线
  ->set_loopback_cb(create_loopback)
  ->seal();

S4U 接口指南

S4U 是 SimGrid 的现代 C++ 接口，从 v3.33 起是主接口。Actor → Activity（Exec/Comm/Io）→ Resource（Host/Link/Disk）。

概念类说明
执行流Actor模拟中独立执行单元
引擎Engine全局单例
通信汇聚Mailbox名字匹配 send/recv，不涉及网络延迟

Mailbox 通信

Mailbox::put(data, size);                       // 阻塞
CommPtr c = Mailbox::put_async(data, size);     // 异步
c->wait(); c->test(); c->wait_for(5.0);
Comm::sendto(src, dst, data, size);             // 直接通信

性能模型

资源模型

网络模型对比

模型特点适用
LV08TCP窗口+慢启动(×13.01)+ACK竞争+bf=0.97+wS=20537通用TCP
CM02仅TCP窗口，可解释性强简单TCP
SMPI分段校正+eager/rendezvous+115种集合算法MPI
Constant固定1秒抽象算法
Raw简单公平共享PCIe总线

LV08 详解

latency-factor=13.01（慢启动）、bandwidth-factor=0.97（TCP头开销）、weight-S=20537（RTT不公平性）、cross-traffic=0.05（ACK反向占用5%带宽）。A→B 100kB/1Mbps/10ms: 简单模型 0.81s → LV08 0.996s

校准

真实平台基准测试 → 测量不同消息大小延迟 → 计算校正因子 → 验证误差。
