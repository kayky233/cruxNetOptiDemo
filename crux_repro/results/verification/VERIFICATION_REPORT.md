## 综合结论

### 1. Crux 机制验证: 通过

在所有 17 个场景中，`crux` 和 `crux_no_compress` 都一致地优于 `random_same`：
- 常规负载: GPU util 提升 **10-16%**
- 极端拥塞: GPU util 提升 **22-32%**
- high-intensity job JCT 降低 **15-28%**

### 2. 优先级级别 K 的影响

- **K=4 是当前规模下的饱和点**，K 再增大没有额外收益
- K=1 时 Crux 退化为仅靠路径选择（gain 仅 ~3%），说明优先级和路径选择两者缺一不可
- K=2 时大部分收益已实现（~9%），K=3→4 趋于饱和

### 3. 收益来源分解（初步）

当前模型下，Crux 的收益可粗略分解为：
- **路径选择 (path selection)**: 占主导。通过将高 intensity job 分散到不同 DSW 路径来减少链路竞争
- **优先级分配 (priority)**: K=1→4 时额外贡献约 7pp gain（3.22% → 10.00%）
- **DAG 压缩 (compression)**: 在常规场景下几乎无损（crux ≈ crux_no_compress），极端拥塞下有约 5pp 损失

### 4. 模型局限性（与现有计划一致）

- GPU/host 参数未产生差异，说明 intra-host 通信竞争未充分建模
- 不同 agg 数量下 Crux 增益变化方向符合直觉（路径越少收益越大），但缺少带宽绝对值校准
- 无法区分 `crux_no_compress` 的无限级别和硬件真实队列行为的差异

### 5. 与真实 SimGrid 结果的一致性

真实 SimGrid C++ 实验显示 `crux`/`crux_no_compress` makespan 降低约 60%，useful GPU fraction 提升约 153%。Python 模拟器的收益量级较小（10-32% util gain），这是因为 Python 模拟器的竞争模型简化为相对权重比较，不包含 SimGrid 的链路排队和资源共享细节。但两者在定性结论上一致：**Crux 策略在高通信竞争场景下有效保护高价值训练作业**。
